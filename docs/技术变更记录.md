# 技术变更记录

## 2025-02-28 - 支持多 Cron 表达式与调度时间调整

### 变更背景

1. 原有配置只支持单个 cron 表达式，无法满足同一任务在多个时间点执行的需求
2. 原定的 8:30 执行时间过晚，需要调整为更合理的 9:40 和 17:00

### 变更内容

#### 1. 数据模型扩展 (src/models.py)

新增 crons 字段支持多个 cron 表达式：

python
# 修改前
cron: str = ""               # Cron 表达式

# 修改后
cron: str = ""               # Cron 表达式（单个）
crons: Optional[List[str]] = None  # Cron 表达式列表（多个执行时间）


#### 2. 调度器重构 (src/scheduler.py)

- 新增 _parse_cron() 方法解析 cron 表达式
- 重构 add_task() 方法支持处理多个 cron
- 多个 cron 时自动为 job id 添加后缀避免冲突
- 更新启动日志显示任务数量和定时任务数量

#### 3. 任务配置更新 (tasks/douying-collect.yaml)

修改前：
cron: "30 8 * * 1-5"  # 周一到周五每天 8:30

修改后：
crons:
  - "40 9 * * *"   # 每天 9:40
  - "0 17 * * *"   # 每天 17:00

### 影响范围

| 文件 | 修改内容 |
|------|----------|
| src/models.py | 添加 crons 字段支持多执行时间 |
| src/scheduler.py | 重构 add_task 方法，新增 _parse_cron 方法 |
| tasks/douying-collect.yaml | 调度时间改为每天 9:40 和 17:00 |
| docs/部署指南.md | 更新配置示例说明多 cron 用法 |

### 向后兼容性

- 保留原有的 cron 字段，继续支持单个 cron 表达式
- crons 字段为可选，未指定时使用 cron 字段
- 现有配置无需修改即可正常工作

### 使用示例

# 单个执行时间（兼容旧配置）
cron: "0 2 * * *"

# 多个执行时间（新功能）
crons:
  - "0 9 * * *"    # 每天 9:00
  - "0 18 * * *"   # 每天 18:00
  - "0 22 * * *"   # 每天 22:00

## 2025-02-26 - 服务部署问题修复

### 问题背景

初次部署时服务无法正常启动，状态一直是 SERVICE_PAUSED。

### 根本原因

1. **虚拟环境路径问题**：Windows 服务默认使用 LocalSystem 账户运行，无法访问用户目录下的虚拟环境 (.venv)
2. **NSSM 参数传递错误**：install.bat 中参数传递方式不正确，导致 -m src.scheduler 无法被正确识别

### 修复方案

#### 1. 切换到全局 Python

修改前：
set PYTHON_EXE=%PROJECT_ROOT%\.venv\Scripts\python.exe

修改后：
set PYTHON_EXE=C:\python314\python.exe

原因：全局安装的 Python (C:\Python314\python.exe) 对所有用户账户可访问，包括 LocalSystem。

#### 2. 修复 NSSM 参数传递

修改前：
nssm install PyTaskSched %PYTHON_EXE% -m src.scheduler

修改后：
nssm install PyTaskSched %PYTHON_EXE%
nssm set PyTaskSched AppParameters "-m src.scheduler"

原因：NSSM 需要分别设置 Application 和 AppParameters，不能在 install 命令中直接传递参数。

### 修改的文件

| 文件 | 修改内容 |
|------|----------|
| scripts\install.bat | Python 路径改为全局版本，修复参数传递方式 |
| scripts\uninstall.bat | 服务名更新为 PyTaskSched |
| README.md | 服务名更新为 PyTaskSched |

### 部署要求变更

之前：支持使用虚拟环境 Python

现在：必须使用全局安装的 Python
- Python 必须安装到 C:\Python314
- 安装时必须勾选 "Install for all users"
- 依赖必须安装到全局 Python

### 经验总结

1. **Windows 服务开发原则**：
   - 服务运行在 LocalSystem 账户下
   - LocalSystem 无法访问用户目录 (C:\Users\...)
   - 所有依赖必须安装在系统级目录

2. **NSSM 使用技巧**：
   - Application: 可执行文件路径
   - AppParameters: 传递给程序的参数（单独设置）
   - AppDirectory: 工作目录（重要！）

3. **调试方法**：
   - 查看错误日志：logs\scheduler_stderr.log
   - 手动测试命令：python -m src.scheduler --list
   - 检查服务配置：nssm get PyTaskSched <parameter>

## 2025-02-27 - Playwright 路径配置与任务调度优化

### 问题背景

抖音采集任务在 Windows 服务模式下运行时，Playwright 无法找到浏览器可执行文件，导致任务失败。

### 根本原因

Windows 服务默认使用 LocalSystem 账户运行，Playwright 在 C:\WINDOWS\system32\config\systemprofile\AppData\Local\ms-playwright 下查找浏览器，但浏览器实际安装在当前用户目录。

### 解决方案

在任务配置中添加 PLAYWRIGHT_BROWSERS_PATH 环境变量，指向当前用户的 Playwright 安装路径。

修改前：
env: {}

修改后：
env:
  PLAYWRIGHT_BROWSERS_PATH: "C:\Users\Administrator\AppData\Local\ms-playwright"

### 其他优化

1. **禁用重试机制**：将 max_retries 从 3 改为 0，避免任务失败时重复执行
2. **调整调度时间**：从每小时执行改为周一到周五每天 8:30 执行

修改后的 cron：
cron: "30 8 * * 1-5"  # 周一到周五每天 8:30

### 影响范围

| 文件 | 修改内容 |
|------|----------|
| tasks/douying-collect.yaml | 添加环境变量，禁用重试，调整 cron |

### 验证结果

任务成功执行，Cookie 验证通过，采集到 4 个视频（全部跳过-已存在）。
